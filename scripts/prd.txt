# W3Uptime On-Chain Governance Transparency Feature
# Product Requirements Document (PRD)

---

## Problem Statement

W3Uptime currently has a centralized governance system where users create proposals, vote, and comment on community decisions. However, there is no verifiable proof that votes are counted accurately or that governance decisions are authentic. This lack of transparency undermines trust in what should be a trustless Web3 platform.

### Pain Points
- Users cannot verify their votes were counted correctly
- No permanent, immutable record of governance decisions
- Centralized database can be modified without detection
- No cryptographic proof of vote authenticity
- Trust relies entirely on platform integrity

### Target Users
- **Validators**: Active participants who monitor websites and want voice in platform governance
- **Monitor Owners**: Users who pay for monitoring services and want transparent decision-making
- **Community Members**: All users who want verifiable, trustworthy governance

### Success Metrics
- 100% of proposals recorded on-chain with verifiable transaction hashes
- All votes cryptographically signed and batch-verified on-chain
- Public transparency dashboard with Sepolia Etherscan links for all governance actions
- Users can independently verify their vote inclusion in on-chain tally

## Capability Tree

### Capability: Smart Contract Foundation
Core blockchain infrastructure for governance transparency on Sepolia testnet.

#### Feature: Governance Smart Contract
- **Description**: Solidity contract that records proposals and finalizes votes on Sepolia blockchain
- **Inputs**: Proposal content hash, voter signatures, vote types
- **Outputs**: On-chain events (ProposalCreated, ProposalFinalized), immutable state
- **Behavior**: Store proposal metadata, verify EIP-712 signatures in batch, tally votes, emit events

#### Feature: EIP-712 Typed Data Definitions
- **Description**: Define structured data types for gasless signature-based voting
- **Inputs**: Domain separator (contract address, chain ID), vote message structure
- **Outputs**: TypedDataDomain and TypedDataField definitions for ethers.js
- **Behavior**: Enable users to sign human-readable vote messages without gas fees

#### Feature: Contract Deployment Scripts
- **Description**: Hardhat/Foundry scripts to deploy and verify contract on Sepolia
- **Inputs**: Deployer private key, constructor parameters
- **Outputs**: Deployed contract address, verified source on Etherscan
- **Behavior**: Deploy contract, verify source code, output deployment artifacts

---

### Capability: Database Schema Migration
Extend existing Prisma models to support on-chain governance tracking.

#### Feature: Proposal Model Extensions
- **Description**: Add on-chain fields to existing Proposal model for blockchain tracking
- **Inputs**: Existing Proposal table structure
- **Outputs**: Extended model with onChainId, contentHash, txHash fields
- **Behavior**: Migration adds nullable fields (backward compatible), indexes for efficient queries

#### Feature: Signed Vote Model
- **Description**: New model to store EIP-712 signatures for off-chain voting
- **Inputs**: User wallet address, proposal ID, vote type, signature
- **Outputs**: SignedVote records with signature validation status
- **Behavior**: Store signatures until finalization, track which signatures submitted on-chain

#### Feature: Governance Status Enum
- **Description**: Define on-chain proposal lifecycle states
- **Inputs**: None
- **Outputs**: OnChainStatus enum (DRAFT, PENDING_ONCHAIN, ACTIVE, PASSED, FAILED, FINALIZED)
- **Behavior**: Track proposal state from creation through finalization

---

### Capability: On-Chain Proposal Creation
Users pay gas to record proposals on Sepolia blockchain for permanent verification.

#### Feature: Proposal Content Hashing
- **Description**: Generate keccak256 hash of proposal title and description
- **Inputs**: Proposal title (string), description (string)
- **Outputs**: bytes32 content hash
- **Behavior**: Concatenate title+description, compute keccak256 hash for on-chain storage

#### Feature: Proposal Creation Transaction
- **Description**: API endpoint to create proposal and record on-chain
- **Inputs**: Authenticated user, proposal data, signed transaction from user wallet
- **Outputs**: Proposal record in database, transaction hash, block number
- **Behavior**: User signs tx with MetaMask, frontend submits to chain, backend records result

#### Feature: Transaction Confirmation Listener
- **Description**: Listen for ProposalCreated events and update database
- **Inputs**: Contract address, event filter
- **Outputs**: Updated Proposal records with confirmed on-chain status
- **Behavior**: Poll or subscribe to events, update proposal status on confirmation

---

### Capability: Gasless Signature-Based Voting
Users sign votes off-chain (free) using EIP-712 typed data.

#### Feature: Vote Signature Request
- **Description**: Generate EIP-712 typed data for user to sign
- **Inputs**: Proposal ID, vote type (UPVOTE/DOWNVOTE), user wallet address
- **Outputs**: TypedData object for wallet signing
- **Behavior**: Construct domain separator, message struct, return for signing

#### Feature: Signature Submission API
- **Description**: Receive and validate user vote signatures
- **Inputs**: Proposal ID, vote type, EIP-712 signature, user wallet address
- **Outputs**: SignedVote record, updated vote counts in UI
- **Behavior**: Verify signature recovers to claimed address, store signature, update real-time counts

#### Feature: Vote Change Support
- **Description**: Allow users to change their vote during voting period
- **Inputs**: New vote signature from same user
- **Outputs**: Updated SignedVote record with new signature
- **Behavior**: Replace old signature, update vote counts, invalidate previous signature

---

### Capability: Automated Proposal Finalization
Platform batches signatures and submits final results on-chain when voting ends.

#### Feature: Voting Period Enforcement
- **Description**: Track 7-day voting period and prevent votes after deadline
- **Inputs**: Proposal creation timestamp
- **Outputs**: votingEndsAt timestamp, isVotingActive boolean
- **Behavior**: Calculate end time (creation + 7 days), reject votes after deadline

#### Feature: Quorum Calculation
- **Description**: Check if 60% of platform users participated in voting
- **Inputs**: Total user count, unique voters count
- **Outputs**: Boolean hasQuorum, participation percentage
- **Behavior**: Query total eligible users, compare to voter count, return quorum status

#### Feature: Batch Signature Aggregation
- **Description**: Collect all valid signatures for on-chain submission
- **Inputs**: Proposal ID
- **Outputs**: Arrays of addresses, vote types, signatures for contract call
- **Behavior**: Query SignedVote records, format for contract finalizeProposal function

#### Feature: Finalization Transaction
- **Description**: Platform wallet submits batch to smart contract
- **Inputs**: Aggregated signatures, platform signer private key
- **Outputs**: Finalization transaction hash, updated proposal status
- **Behavior**: Call contract.finalizeProposal(), wait for confirmation, update database

#### Feature: Finalization Scheduler
- **Description**: Automated job to finalize proposals when voting ends
- **Inputs**: Current timestamp
- **Outputs**: Finalized proposals
- **Behavior**: Query proposals where votingEndsAt <= now AND onChainStatus = ACTIVE, trigger finalization

---

### Capability: Transparency Dashboard
Public page showing all proposals with blockchain verification links.

#### Feature: Proposal List with On-Chain Status
- **Description**: Display all governance proposals with blockchain verification
- **Inputs**: None (public endpoint)
- **Outputs**: List of proposals with tx hashes, status, vote breakdowns
- **Behavior**: Query proposals, include Sepolia Etherscan links for creation and finalization txs

#### Feature: Individual Proposal Verification
- **Description**: Allow users to verify their vote was included in on-chain tally
- **Inputs**: User wallet address, proposal ID
- **Outputs**: Boolean voteIncluded, signature used, on-chain proof
- **Behavior**: Check if user's signature was in finalization batch, provide verification data

#### Feature: Governance Data Export API
- **Description**: API endpoint for independent verification of governance data
- **Inputs**: Optional date range, proposal IDs
- **Outputs**: JSON export of all governance data with tx hashes
- **Behavior**: Return proposal data, vote counts, signatures, tx hashes for external verification

#### Feature: Real-Time Vote Display
- **Description**: Show current vote counts with live updates
- **Inputs**: Proposal ID
- **Outputs**: Upvote count, downvote count, participation percentage
- **Behavior**: Query SignedVote counts, display with progress bar toward quorum

- **Maps to capability**: Smart Contract Foundation

```
    /------------\
   /  Unit Tests  \ <- 60% (Business logic, utilities)
  /----------------\
```

### Coverage Requirements
- Line coverage: 80% minimum
- Branch coverage: 75% minimum
- Function coverage: 90% minimum

### Critical Test Scenarios

#### Smart Contract (W3Governance.sol)
**Happy path**:
- Create proposal emits correct event with content hash
- Finalize proposal with valid signatures updates state correctly
- Vote counts match submitted signatures

**Edge cases**:
- Empty signature array in finalization
- Single voter proposals
- Exactly 60% quorum boundary
- Exactly 2/3 majority boundary

**Error cases**:
- Invalid signature rejected
- Duplicate signature rejected
- Finalization before voting period ends
- Non-existent proposal ID

**Integration points**:
- Contract events parsed correctly by backend
- Gas estimation accurate for batch sizes

#### Signature Validation
**Happy path**:
- Valid EIP-712 signature accepted
- Correct wallet address recovered

**Edge cases**:
- Very long proposal descriptions
- Special characters in proposal content

**Error cases**:
- Malformed signature rejected
- Wrong chain ID rejected
- Expired signature rejected

#### Finalization Flow
**Happy path**:
- Proposal with quorum passes correctly
- Results match off-chain counts

**Edge cases**:
- Large batch (100+ signatures)
- Network congestion scenarios

**Error cases**:
- Insufficient platform wallet balance
- Network failure during submission

```solidity

    uint256 id;
    address proposer;
    bytes32 contentHash;
    uint256 createdAt;
    uint256 votingEndsAt;
    uint256 upvotes;
    uint256 downvotes;
    bool finalized;
```

### Technology Decisions

**Decision: EIP-712 Typed Data for Voting**
- **Rationale**: Human-readable signatures, gasless UX, standard wallet support
- **Trade-offs**: More complex than simple message signing, requires domain separator
- **Alternatives considered**: Plain message signing (less user-friendly), on-chain voting (gas costs)

**Decision: Batch Signature Verification**
- **Rationale**: Single transaction for all votes reduces gas, atomic finalization
- **Trade-offs**: Larger transaction size, potential gas limit issues with many voters
- **Alternatives considered**: Individual vote submission (expensive), Merkle proofs (complex)

**Decision: Platform Pays Finalization Gas**
- **Rationale**: Ensures proposals are finalized regardless of creator's ETH balance
- **Trade-offs**: Platform bears operational cost
- **Alternatives considered**: Creator pays (may abandon proposals), DAO treasury (complex)

**Decision: 7-Day Voting Period**
- **Rationale**: Sufficient time for community participation, standard governance period
- **Trade-offs**: Slower decision making
- **Alternatives considered**: Shorter periods (less participation), variable periods (complex)

## Risks and Mitigations

### Technical Risks

**Risk: Gas Costs Exceed Budget for Large Batches**
- **Impact**: High - Could fail to finalize popular proposals
- **Likelihood**: Medium
- **Mitigation**: Implement batch size limits, multiple finalization transactions if needed
- **Fallback**: Manual finalization in smaller batches

**Risk: Signature Replay Attacks**
- **Impact**: High - Could corrupt vote counts
- **Likelihood**: Low (mitigated by design)
- **Mitigation**: Include proposalId and nonce in signed message, verify on-chain
- **Fallback**: Contract pause mechanism

**Risk: EIP-712 Wallet Compatibility**
- **Impact**: Medium - Some users may not be able to vote
- **Likelihood**: Low (standard supported by major wallets)
- **Mitigation**: Test with MetaMask, WalletConnect, Coinbase Wallet
- **Fallback**: Provide plain message signing alternative

### Dependency Risks

**Risk: Sepolia Network Issues**
- **Impact**: Medium - Temporary inability to create/finalize proposals
- **Likelihood**: Low
- **Mitigation**: Graceful error handling, retry logic
- **Fallback**: Queue operations for later submission

**Risk: Platform Wallet Key Compromise**
- **Impact**: Critical - Could submit fraudulent finalizations
- **Likelihood**: Low with proper security
- **Mitigation**: Use secure key management (AWS KMS, hardware wallet)
- **Fallback**: Contract pause, key rotation mechanism

### Scope Risks

**Risk: Feature Creep in Governance Rules**
- **Impact**: Medium - Delayed delivery
- **Likelihood**: Medium
- **Mitigation**: Lock requirements: 60% quorum, 2/3 majority, 7 days. No changes.
- **Fallback**: Defer additional rules to future iteration

**Risk: Integration with Existing System**
- **Impact**: Medium - May require more refactoring
- **Likelihood**: Medium
- **Mitigation**: Use backward-compatible nullable fields, gradual migration
- **Fallback**: Create separate governance system if integration too complex

### Voting Rules Summary
- **Voting Period**: 7 days from proposal creation
- **Quorum Requirement**: At least 60% of platform users must participate
- **Passing Threshold**: 2/3 (66.67%) of votes must be upvotes
- **Vote Changes**: Allowed during voting period (new signature replaces old)
